# KorriPay Settlement Certificate Standard (v1.0)

This document standardizes the specifications, cryptographic requirements, and visualization structures for **Settlement Certificates** issued by the KorriPay settlement platform.

---

## 1. Overview
KorriPay has transitioned from consumer payment receipts to formal, enterprise-grade **Settlement Certificates**. A Settlement Certificate serves as a cryptographically verifiable proof of a finalized settlement instruction across the GIWA L2 network.

---

## 2. Standard Schema Fields (14 Fields)
Every Settlement Certificate strictly displays 14 canonical fields, partitioned into 4 distinct groups:

### 1. Settlement Identity
*   **Settlement ID:** The unique platform identifier of the settlement transaction (e.g. `settlement-1718928374921`).
*   **Settlement Status:** The current state of the settlement lifecycle: `Settled`, `Success`, `Pending`, or `Failed`.
*   **Settlement Timestamp:** The ISO-8601 timestamp representing when the transaction was completed.
*   **Settlement Duration:** The time delta in seconds from request generation to L2 block finality.

### 2. On-Chain Ledger Record
*   **Transaction Hash:** The transaction hash generated by the sequencer on the GIWA L2 network.
*   **L2 Block Number:** The block index where the transaction is recorded on the L2 state.
*   **Gas Used:** The exact execution gas units consumed by the settlement precompile contract.
*   **Confirmation Count:** The depth of confirmation blocks. For finalized blocks, it displays `640+ (Deep Finalized)`.

### 3. Compliance & Attestations
*   **Compliance Screening:** The automated compliance score and status (e.g. `Passed (Zero-Knowledge Compliance Passport Verified, Low Risk)`).
*   **Attestation References:** The Ethereum Attestation Service (EAS) UID reference mapping to the credential on-chain.
*   **Proof Integrity Status:** The status of the zero-knowledge validation (e.g. `Valid ZK-SNARK Cryptographic Proof Attested`).

### 4. Infrastructure & Protocol
*   **GIWA Network:** The name of the connected L2 network (e.g. `GIWA Testnet (Sepolia)`).
*   **Explorer Link:** A direct link to inspect the raw transaction on the GIWA L2 block explorer.
*   **Protocol Version:** The active hardfork version and client node details (e.g. `Karst Hardfork (Osaka EVM) / op-reth`).

## 3. Formats & Export Channels

### 3.1 Interactive Certificate Modal
Integrated directly within the **Settlement Ledger** (`dashboard.html`):
*   Clicking a transaction row hydates the `#modal-proof` with the 14-field layout.
*   Shows a visual timeline of the 9-stage lifecycle matching real state progression.
*   Features active controls to:
    1.  *View Explorer* — Open transaction hash on the L2 block explorer.
    2.  *Download PDF* — Print an institutional-grade certificate.
    3.  *Export JSON* — Download the machine-readable schema.
    4.  *Verify ZK Proof* — Simulate verification of precompile MODEXP and P256 constraints.

### 3.2 PDF Certificate (Print Layout)
*   Rendered as an A4-styled Portrait decree with a formal letterhead, verification seal, partitioned data tables, full 9-stage lifecycle checklist, and signature block.
*   Features print-specific media queries to hide buttons and adjust layout scales.
